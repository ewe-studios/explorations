FS Handler — Native Filesystem Operations

The fs_handler provides native Rust filesystem operations inside the VM — no child processes, no shell-outs.

Architecture

Source: fs_handler.rs:7-22

flowchart LR A[ShellMessage::FsRequest] --> B[fs_handler dispatcher] B --> C{Operation type?} C -->|read| D[ops::read] C -->|write| E[streaming::write] C -->|mkdir| F[ops::mkdir] C -->|rmdir| G[ops::rmdir] C -->|unlink| H[ops::unlink] C -->|chmod| I[ops::chmod] C -->|grep| J[ops::grep: native regex] C -->|sed| K[ops::sed: native regex] C -->|tree| L[ops::tree: walkdir] D --> M[FsResult] E --> M F --> M G --> M H --> M I --> M J --> M K --> M L --> M

Aha: The fs_handler is sync-only — no tokio, no async fn, no AsyncRead/AsyncWrite. All I/O uses std::fs and std::io. This keeps the binary cross-compilable to linux-musl without a tokio runtime.

Streaming Writes

Source: fs_handler/streaming.rs (308 lines)

sequenceDiagram participant Client as Host client participant Handler as fs_handler participant Temp as Temp file participant Target as Target file Client->>Handler: FsChunk (data part 1) Handler->>Temp: Write data Client->>Handler: FsChunk (data part 2) Handler->>Temp: Write data Client->>Handler: FsEnd Handler->>Target: rename(temp, target) Target-->>Client: Success

Streaming writes use temp files with UUID-based naming to avoid partial writes on disk:

  1. Receive FsChunk frames with correlation ID
  2. Write to temp file
  3. On FsEnd, rename temp file to target (atomic on same filesystem)
  4. On error, delete temp file
