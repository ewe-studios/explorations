iii-network — Userspace TCP/IP for VM Sandboxes

iii-network provides a userspace TCP/IP stack for iii worker VM sandboxes using smoltcp, bridging guest ethernet frames through shared memory to proxy tasks.

What It Does

flowchart TB subgraph GuestVM["Guest VM"] G1["Applications"] G2["virtio-net device"] end subgraph Host["Host Process"] subgraph NetWorker["libkrun NetWorker thread"] NW1["rx_ring: guest → host frames"] NW2["tx_ring: host → guest frames"] end subgraph PollThread["smoltcp poll thread (dedicated OS thread)"] S1["SmoltcpDevice: read rx_ring"] S2["smoltcp Interface + SocketSet"] S3["ConnectionTracker"] S4["DNS Interceptor"] S5["UDP Relay"] end subgraph ProxyTasks["tokio proxy tasks"] P1["TCP proxy: guest ↔ host"] end end subgraph External["External Network"] E1["Host DNS resolver"] E2["Host services"] end G1 --> G2 G2 --> NW1 NW1 --> S1 S1 --> S2 S2 --> S3 S2 --> S4 S2 --> S5 S3 --> P1 S4 --> E1 S5 --> E2 P1 --> E2 P1 --> S2 S2 --> NW2 NW2 --> G2 G2 --> G1

Aha: The entire TCP/IP stack runs in userspace via smoltcp — no kernel networking, no TUN/TAP devices, no root privileges needed. This is critical for sandboxed VMs where the host doesn't want to expose its network stack.

Key Design Decisions

DecisionWhy
smoltcp in userspaceNo kernel networking, no root, full sandbox isolation
Dedicated OS thread for poll loopsmoltcp is sync-only, no async runtime
Pre-inspection of framesCreate TCP sockets before smoltcp sees SYN (prevents auto-RST)
DNS interceptorHijack guest DNS queries, resolve on host
UDP relayHandle non-DNS UDP outside smoltcp
