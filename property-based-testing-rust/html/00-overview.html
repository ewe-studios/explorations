Property-Based Testing in Rust — Overview

What property-based testing is

Property-based testing (PBT) shifts your testing mindset from examples to invariants.

In traditional example-based testing, you write individual test cases:

#[test]
fn test_reverse() {
    assert_eq!(reverse(&vec![1, 2, 3]), vec![3, 2, 1]);
    assert_eq!(reverse(&vec![]), vec![]);
    assert_eq!(reverse(&vec![42]), vec![42]);
}

In property-based testing, you state a general truth about your code, and the framework generates thousands of inputs to try to falsify it:

proptest! {
    #[test]
    fn test_reverse(ref input in any::<Vec<i32>>()) {
        let reversed = reverse(&input);
        let double_reversed = reverse(&reversed);
        prop_assert_eq!(double_reversed, input);  // invariant: reverse(reverse(x)) == x
    }
}

The framework generates random vectors of integers — empty, single-element, huge, negative, zero, sorted, reverse-sorted, and every weird shape it can think of — and verifies your invariant holds for all of them.

Why property-based testing matters

Example tests miss edge cases

When you write example-based tests, you test what you already think of. Properties force you to test what you didn't. The framework's random generator explores combinations you would never have written by hand: vectors with 10,000 elements, strings with embedded null bytes, deeply nested structures, boundary values at exactly i32::MIN and i32::MAX.

Properties document intent

A test case says "this specific input produces this specific output." A property says "this is what my code guarantees, regardless of input." The property is a specification — readable, verifiable, and impossible to drift out of sync with the implementation.

Shrinking saves debugging time

When a property-based test finds a failing input, the framework automatically shrinks it — repeatedly tries smaller inputs until it finds the minimal failing case. Instead of a crash on a 500-element vector, you get a clean report on vec![3, -1]. This eliminates hours of manual reproduction.

Coverage scales with time

Run an example test once — it passes or fails. Run a property test — it generates thousands of inputs. Run it longer — it generates more. The proptest default is 256 successful cases per property; you can increase it to 100,000 in CI or 1,000,000 overnight. More time = more confidence.

The three core concepts

Properties

A property is an assertion about your code that must hold for all valid inputs. Examples:

  • Idempotence: compress(decompress(x)) == x for all x
  • Round-trip: parse(format(x)) == Some(x) for all x
  • Inverse: encode(decode(x)) == x for all x
  • Consistency: Two implementations produce the same output for all inputs
  • Error handling: Invalid inputs always return Err, never panic
  • Monotonicity: Adding constraints never increases the set of valid instances

Generators

A generator (Strategy in proptest) produces random values of a specific type. proptest provides generators for primitives (any::()), collections (vec(any::(), 0..10)), and combinators (prop_oneof![..., ...]). You build generators for your own types by combining these primitives.

Assertions

Property assertions are like regular assertions but with shrinking support. When they fail, proptest finds the smallest input that still fails. Use prop_assert!, prop_assert_eq!, prop_assert_ne! instead of assert!, assert_eq!, assert_ne!.

Property-based testing is not fuzzing

Property-based testing and fuzzing are complementary:

AspectProperty-Based TestingFuzzing
Input typeTyped, generated by strategiesUntyped, raw bytes
CheckUser-defined invariants (assertions)Crashes, panics, sanitizer violations
GenerationSmart generators with semantic knowledgeCoverage-guided mutations
ShrinkingDeep, multi-pass automatic shrinkingBasic minimization
Best forLogic correctness, round-trips, consistencyMemory safety, panic-free guarantees
RuntimeFast per case, limited cases (256 default)Slower per case, massive cases (millions)
CoverageDepends on generatorsEdge-driven, systematic

Use property tests to verify your code's logic: round-trips, invariants, consistency between implementations.

Use fuzzing to verify your code's safety: no panics, no memory corruption, no undefined behaviour on arbitrary input.

Both should be in your test suite.

When to use property-based testing

Always use for:

  • Parsing/formatting round-trips: format(parse(x)) must recover x
  • Compression/decompression: decompress(compress(x)) must equal x
  • Serialization/deserialization: to_json(from_json(x)) must recover x
  • Inverse operations: decode(encode(x)) must equal x
  • Ordering and sorting: sort(sort(x)) == sort(x) (idempotence)
  • Constraint satisfaction: valid inputs satisfy all constraints
  • Consistency between two implementations

Consider for:

  • Any function with multiple branches — properties explore all paths
  • Complex data transformations — properties catch edge cases
  • Input validation — properties verify that invalid inputs are rejected

Skip for:

  • UI rendering, graphics, or anything with visual output
  • Code with external side effects (network, filesystem) — use mocks
  • Performance benchmarks — use criterion
