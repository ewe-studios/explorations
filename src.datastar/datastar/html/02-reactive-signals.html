Datastar -- Reactive Signals

The signal system in engine/signals.ts (781 lines) is the heart of Datastar. It implements fine-grained reactivity: instead of re-rendering a virtual DOM on every change, individual DOM nodes update only when the specific signals they depend on change.

Aha: Datastar's signal system is built from interfaces and module-level state, not classes. ReactiveNode and Link are pure interfaces тАФ all the state lives in module-level variables (activeSub, version, batchDepth, etc.) and the actual signal objects are minimal records with underscore-prefixed fields. Operations like signal(), computed(), and effect() return bound functions, not node instances. This keeps the runtime overhead per signal extremely low.

Core Interfaces

ReactiveNode

The fundamental building block. Every signal, computed value, and effect is a ReactiveNode тАФ but it's an interface, not a class:

Aha: ReactiveNode has no value_, no getter, no fn_. It's a pure graph node тАФ just edges and state. Link

A directed edge connecting a source (dependency) to a consumer (subscriber):

// engine/signals.ts:22-30
interface Link {
  version_: number          // Global version at link creation
  dep_: ReactiveNode        // The source (producer)
  sub_: ReactiveNode        // The consumer
  prevSub_?: Link           // Previous link in source's sub list
  nextSub_?: Link           // Next link in source's sub list
  prevDep_?: Link           // Previous link in consumer's dep list
  nextDep_?: Link           // Next link in consumer's dep list
}

Aha: One object serves as a node in TWO doubly-linked lists simultaneously. This is the key insight. A single Link object is both:

  1. A node in dep_.subs_ тАФ the subscription list of the source signal
  2. A node in sub_.deps_ тАФ the dependency list of the consumer

This means there's no duplication тАФ one Link object maintains four pointers (prevSub_, nextSub_, prevDep_, nextDep_) that stitch it into two independent doubly-linked lists. When the link is removed, it must be unspliced from both lists at once.

The version_ field тАФ stale link detection:

Each time startTracking() is called, the global version counter increments. When link() creates or reuses a link, it sets link.version_ = version. During endTracking(), any link whose version_ is older than the current global version is stale тАФ the dependency was not read during this evaluation and should be removed. This avoids a full rebuild of the dependency graph on every re-evaluation.

Stack тАФ Linked-List Stack (Not an Array)

// engine/signals.ts:32-35
interface Stack<T> {
  value_: T
  prev_?: Stack<T>
}

Aha: This is NOT an array-based stack тАФ it's a singly-linked list. Each push creates a new Stack object with a prev_ pointer to the previous top. This avoids array allocation and resizing in the tight propagation loop.

Push:

stack = { value_: next, prev_: stack }

Creates a new Stack node whose prev_ points to the current top, then replaces stack. O(1), no allocation beyond the single object.

Pop:

link = stack.value_!
stack = stack.prev_

Extracts value_, then moves stack to prev_. The old Stack node becomes garbage.

Used in two places:

  1. propagate() тАФ Stack saves sibling subscribers when descending into a node's subscription chain. If a signal has subscribers A, B, C, the algorithm descends into A first and saves B and C on the stack. After A's branch is fully processed, it pops B and continues.
  1. checkDirty() тАФ Stack saves the caller's position when recursively descending into a dependency's own dependencies. After checking if a deeper source is dirty, it pops back to continue checking sibling dependencies.

The undefined in Stack for propagate() allows pushing a "null" slot тАФ when the current node has only one subscriber (no siblings), the stack gets { value_: undefined, prev_: stack } to maintain the depth counter, and when popped, the if (link) guard skips it.

Why queuedEffects has undefined slots:

When flush() starts executing an effect, it sets the slot to undefined. This prevents stale references тАФ if the effect re-queues itself during execution, it goes into a new slot. The undefined value also allows the while (notifyIndex < queuedEffectsLength) loop to skip consumed slots.

The peeking stack:

prevSub and activeSub form a two-element stack for tracking contexts:

startPeeking(newSub):  prevSub = activeSub; activeSub = newSub   // push
stopPeeking():         activeSub = prevSub; prevSub = undefined  // pop

This is NOT a general-purpose stack тАФ it only supports depth 1 nesting. If you call startPeeking() twice without stopPeeking(), the first context is lost. But this is intentional: startPeeking is always paired with stopPeeking in a try/finally block, so nesting is safe in practice.

Batching

Nested batches increment batchDepth. Only when the outermost batch ends (batchDepth reaches 0) do effects flush and patches dispatch. This coalesces multiple signal changes into a single propagation cycle.

The Four-Phase Cycle

Every signal change goes through four phases:

flowchart LR A["signal(value)"] --> B["propagate()\nmark Dirty+Pending"] B --> C["notify()\nqueue effect"] C --> D["flush()\nrun effect"] D --> E["endTracking()\nclean stale deps"] E -.-> B
  1. SET тАФ signalOper detects a value change, calls propagate()
  2. MARK тАФ propagate() walks the sub graph, setting Dirty/Pending flags
  3. QUEUE тАФ notify() puts Watching effects into the queue
  4. RUN тАФ flush() executes each effect, which re-evaluates, calls endTracking() to clean up

The complexity lives in phases 2 and 4 тАФ walking the graph correctly and handling diamond dependencies.

Why: During re-evaluation, signals are typically read in the same order. If the dep was just added, it's at the tail. This is O(1) for the common case.

Why: When a computed re-evaluates and reads its deps in the same order, they'll appear sequentially. This moves the tail forward without creating a new link тАФ the version bump marks it as "still valid this round."

Why the subs_ chain for queuing? This is an optimization. When multiple effects are linked (e.g., a parent effect containing a child effect), only the LEAF effect gets pushed to the array.
