Seed Corpus and Dictionaries

Seed corpus

The seed corpus is a set of initial inputs that the fuzzer uses as starting points. Without seeds, libFuzzer starts from random bytes and discovers code paths slowly. With good seeds, it jumps directly to interesting code.

Where seeds live

By default, libFuzzer looks for seeds in two locations relative to the fuzz target:

  1. corpus// — Grows during fuzzing. New inputs that reach new code paths are saved here automatically.
  2. seeds// — Static seeds you provide. These are read at startup and used alongside the growing corpus.
fuzz/
├── corpus/
│   └── builder/
│       ├── 0123456789abcdef  # Auto-generated during fuzzing
│       └── 1234567890abcdef  # Auto-generated
├── seeds/
│   └── builder/
│       ├── simple.json       # You provide these
│       ├── nested.json       # You provide these
│       └── with_ref.json     # You provide these

Choosing seed inputs

A good seed corpus covers the main entry points and data shapes of your code:

  • Empty input — "", {}, []
  • Minimal valid input — {"type": "string"}, {"minimum": 0}
  • Nested structures — Objects with objects, arrays with objects
  • Reference patterns — {"$ref": "#/$defs/item"} with $defs
  • Edge cases — Maximum nesting depth, maximum array length, boundary values
  • Known trouble spots — Inputs that previously caused crashes

Managing the corpus

The corpus grows during fuzzing. Periodically review and trim it:

# View corpus statistics
cargo +nightly fuzz run builder -- -help=1 2>&1 | grep corpus

# Minimize corpus (remove redundant inputs)
cargo +nightly fuzz run builder -- -merge=1 corpus/builder/

# Run on a specific corpus directory
cargo +nightly fuzz run builder corpus/builder/

Do not delete corpus files arbitrarily. Each one represents a unique code path. Instead, use -merge to remove redundant entries.

Fuzzing dictionaries

A dictionary guides libFuzzer's mutation engine. Instead of random bit flips on individual bytes, the fuzzer can substitute whole tokens — JSON keywords, structural characters, and common patterns. This dramatically improves coverage for structured input formats.

Dictionary file format

A dictionary file is a plain text file with keyword=value pairs:

# JSON Schema keywords
type="type"
properties="properties"
required="required"
allOf="allOf"
anyOf="anyOf"
oneOf="oneOf"
not="not"
if="if"
then="then"
else="else"

# Structural tokens
open_obj="{"
close_obj="}"
open_arr="["
close_arr="]"
colon=":"
comma=","

# Type values
string_val="string"
number_val="number"
integer_val="integer"
boolean_val="boolean"
null_val="null"

Using a dictionary

Pass the dictionary to the fuzzer with -dict=PATH:

cargo +nightly fuzz run builder -- -dict=fuzz/dict

Writing a good dictionary

  1. List all keywords — Every string literal that appears in your parser's decision tree should be in the dictionary.
  2. Include structural tokens — {, }, [, ], :, , as quoted strings. These help the fuzzer build valid JSON structure.
  3. Add common values — true, false, null, 0, "", and boundary values for your domain.
  4. Add escape sequences — For string parsing: \", \\, \/, \n, \t, .
  5. Keep it under 4KB — Larger dictionaries slow down mutations. Focus on high-value tokens.
  6. Update after adding features — If you add a new keyword (e.g., unevaluatedProperties), add it to the dictionary.

Dictionary vs seed corpus

AspectDictionarySeed Corpus
PurposeGuides mutationsProvides starting inputs
Format`keyword="value"` pairsRaw input bytes
Size~1-4KB~1KB-1MB
EvolutionStatic (you maintain)Dynamic (grows during fuzzing)
EffectMakes mutations more meaningfulGives the fuzzer a head start

Both are necessary. Seeds alone will get stuck at local optima. A dictionary alone starts from random bytes and reaches them slowly. Together, they give comprehensive coverage quickly.
